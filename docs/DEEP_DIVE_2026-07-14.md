## 0. TL;DR (read this first)

1. **The stack is real and large.** Neo4j holds ~21.4M `PhoneLog`, 4.3M `DashcamEmbedding`, 4.1M `YOLODetection`, 245K `Speaker`, 24.5K `GlobalSpeaker`, and 772 `Trip` nodes.
2. **The documented "knowledge_map" sync engine does not exist.** `config.yaml`'s `knowledge_map:` block and `scripts/knowledge_sync_all.sh` (line 73) call `python3 -m knowledge_map`, but no such module is importable. Real sync code is `scripts/knowledge_harvest*.py` / `shared_hermes_handler.py`.
3. **Speaker identity has no "me" anchor.** `VoiceIdentity{user_id:'scott'}` exists but is tied to a single Speaker (1 edge). Of 24,506 `GlobalSpeaker`s, only **429 are `confirmed`**; 24,077 are `tentative`. There is no property saying "this GlobalSpeaker is Scott."
4. **The Trip↔DashcamClip link is missing.** `Trip` is rich (driver, tracker, start/end location) and links to `PhoneLog`/`LocationEvent`, but `DashcamClip` does **not** relate to `Trip` — so "every 1-min clip in a Trip" cannot currently be resolved in-graph.
5. **Storage is fragmented across NAS1/NAS2/NAS3/NAS4/SSD_4TB** and the migration to NAS4 is in-flight (just committed). The "centralization" goal is not yet achieved; it is aspirational.
6. **The old `runall.sh` is a hardcoded, single-host script** (`cd ~/git/auto-ingest`, fixed passwords, sequential). It cannot be the "same command on every machine."

The rest of this document reviews each area, lists gaps explicitly, and proposes a concrete target architecture.

## 3. Proposed Target Architecture

### 3.1 One command, every machine: `auto-ingest` CLI
A single entrypoint `bin/auto-ingest` (symlinked to `run`) that:
1. **Detects host** via `config.yaml` (`hostname_pattern`) → resolves `fileserver_root`, `neo4j_uri`, mounts.
2. **Detects hardware** → backend profile (see 3.4).
3. **Reports status** (`auto-ingest status`) → reads a **preprocessing manifest** (3.2) and prints, per source, the current step + counts + last run.
4. **Dispatches** subcommands: `ingest`, `diarize`, `link-speakers`, `claim`, `status`, `sync`, `whoami` (speaker anchor tooling).
5. Honors env overrides; never hardcodes passwords (reads `.env` / `~/.netrc` / secret store).

### 3.2 Preprocessing manifest (solves G4, G5)
A small per-source record (stored as JSON + mirrored to Neo4j as `IngestJob` nodes) tracking each media key through stages:

```
key: dashcam/2026-07-14/CLT_1423_F
stages:
  copied:        done @2026-07-14T10:01
  transcribed:   done @2026-07-14T10:05 (whisper medium, device m2)
  diarized:      done @2026-07-14T10:09 (pyannote, 2 spk)
  embedded:      pending
  linked:        pending
  graph_written: pending
owner:           ""            # claimed worker, else empty
claimed_at:      null
```

`auto-ingest status` reads this to show "what preprocessing metadata step we are at." The graph stage writes a **cleaned JSON/CSV** (one per key) that a separate `import` step loads with **dynamic transactions** (`UNWIND $rows AS r ...` batches from 3.1) — replayable and resumable.

### 3.3 Distributed claim mechanism (solves G5, G6)
Extend the existing `/nas/drop` `.job` queue into a **claim protocol**:
1. Coordinator writes `IngestJob` (manifest) with `owner=""`.
2. Any worker with capacity: `claim <key>` → sets `owner=<host>`, `claimed_at=now` (conditional `SET` guarded by `owner=""`, with TTL reaper for stale claims).
3. Worker **copies** source snips/audio to local fast storage (or streams), **processes** with best-fit backend, writes cleaned JSON/CSV + embeds.
4. Worker **commits** metadata back: graph nodes + manifest `graph_written=done`, `owner=""`.
5. Optional: lockfile/`flock` + Neo4j conditional write prevents double-processing.

This directly enables "copy files, process on another machine, commit back the metadata."

### 3.4 Hardware-aware backend routing (solves G7)
`auto-ingest caps` probes once and caches a profile:

| Capability | Detection | Backend used |
|-----------|-----------|--------------|
| Apple Silicon | `platform.processor()=='arm'` + `sysctl` | **MLX** Whisper + ECAPA (mlx-omni / whisper-mlx) |
| NVIDIA GPU | `nvidia-smi` | **CUDA** (faster-whisper, speechbrain/pyannote CUDA) |
| AMD GPU | `rocminfo` | **ROCm** |
| CPU only | fallback | ONNX Runtime (quantized) |

Routing maps task→backend: transcription, diarization, ECAPA embedding, YOLO each get a preferred backend per host. This is what lets "a number of machines with MLX/ONNX/CUDA/ROCm backends" be utilized automatically.

### 3.5 Speaker identity: anchor "me", then scope per Trip (solves G8, G9, G11)
Two concrete additions on top of the existing linker:

**(a) Establish the Scott anchor.** Add to `GlobalSpeaker` a `person_id` / `is_me` property. Bootstrap:
- Take the existing `VoiceIdentity{user_id:'scott'}`; find its linked `Speaker`/`GlobalSpeaker`.
- Use **your own known recordings** (e.g., a small curated `me/` corpus of clips where you are the only speaker) to compute a Scott centroid, then mark the nearest `GlobalSpeaker` (≥ threshold) as `person_id:'scott', is_me:true`.
- All future `link_global_speakers.py` runs propagate `is_me` to any local `Speaker` clustered into that `GlobalSpeaker`.
- Every `SPEAKER_00`/`SPEAKER_01` per clip then resolves: "this local speaker → GlobalSpeaker X → is_me?" giving you a label in every clip.

**(b) Link DashcamClip → Trip (and Trip→Speaker).** Add `:IN_TRIP` from `DashcamClip` to `Trip` (derive Trip from filename/date/time + `LocationEvent`/`PhoneLog` trip windows). Then per Trip you can isolate one person's clips and, via the anchor, know which local speaker is Scott. This is exactly the "1-min clip that isolates a person in a TRIP" graph query you described.

**(c) Quarantine→confirm.** Promote `tentative`→`confirmed` when a GlobalSpeaker accumulates enough weighted evidence *or* is matched to a labeled anchor (you, known contacts). This attacks G11.

### 3.6 The "brain" + MCP server (long-term, solves the recall goal)
A runtime (in the Hermes agent) that:
1. Runs **pattern extractors** over transcripts before/after graph write: detect date/time ("last Tuesday", "at 3pm"), locations, named people, self-references ("I", "my car"). Store as `Mention`/`Entity` nodes (already partially present: `Entity` 14,761, `MENTIONS` 97,799).
2. Exposes an **MCP server** with tools: `query_graph(cypher)`, `find_speaker(name, trip)`, `when_did(scott, <topic>)`, `who_was_with(scott, <date>)`. Each tool wraps a Cypher query; as new queries appear, add a tool + refine the Cypher — the "functions over time" refinement loop you described.
3. Starts from graph-first, vault-second (already the HLD strategy), and upgrades to semantic + temporal reasoning.

### 3.7 Centralize storage (solves G1, G2)
- Pick **one** canonical vault: `NAS4/fileserver/shared-knowledge` (already the `mirror_vault_path`). Make `/home/scott/knowledge` a symlink to a local clone, kept in sync by a single service (not 4 hardcoded SSH pushes).
- Replace `knowledge_sync_all.sh`'s hardcoded IPs with Tailscale hostnames + a service-discovery list in `config.yaml`.
- Implement the missing `knowledge_map` sync as a thin module wrapping the existing `knowledge_harvest*.py` (don't rewrite; wrap), so `python3 -m knowledge_map` works and HLD/LLD match reality.

### 6.3 Known limitation found (G13)
The per-trip **speaker report** works, but it reflects transcription-granular speakers, not segment-granular, because:
- `Segment` has **no `key`/clip property** and no direct `DashcamClip` link.
- `Transcription` nodes are **aggregated** (one node covers a whole day/batch); `HAS_SEGMENT→Segment→SPOKEN_BY` therefore returns corpus-wide counts (e.g. `SPEAKER_00` = 5.1M segments) when joined by key.

So "every 1-minute clip that isolates a person in a TRIP" is achievable at the **trip/clip** level now (edges exist), but **segment-level** attribution within a trip requires a schema fix: store the clip key on `Segment` (or make `Transcription` per-clip). That is the next refinement, per your "efficient way to link the data we have, then refine" plan.

### 6.5 GlobalSpeaker consolidation (Scott fragmented across 15 identities)
Inspection showed **15 separate `GlobalSpeaker`s** all labeled `Scott`/`is_me=true`, each with 1
`Speaker` and only the `VoiceIdentity` seed carrying an ECAPA embedding. That fragmentation is exactly
why the anchor could not grow. Fix: added `--merge` to `scripts/anchor_speaker_me.py`, which redirects
every `Speaker`'s `SAME_PERSON` edge from the 14 extras onto the canonical seed and `DETACH DELETE`s
the empties. Result: **1 canonical Scott `GlobalSpeaker`** (id `1081161525…`) holding 14 `Speaker`s.
Idempotent.

### 6.6 Anchor auto-grows on future linker runs
`link_global_speakers.py` was modified so the `is_me` flag **propagates** from a `GlobalSpeaker` onto
every `Speaker` linked to it:
- `update_existing_gs_with_assignments` (global-prefilter path): when assigning a new local `Speaker`
  to the canonical Scott `GlobalSpeaker`, it now sets `s.is_me=true, s.person_id='scott', s.label='Scott'`.
- `write_clusters_incremental` (new-cluster path): if any member `Speaker` is already `is_me`, the new
  `GlobalSpeaker` and all its `Speaker`s inherit `is_me`.

Net effect: once `link_global_speakers.py` is next run, every clip whose local speaker clusters into the
Scott `GlobalSpeaker` is automatically labeled Scott — no manual re-anchoring needed. (Not run yet; it is
the heavy ECAPA job and should be launched deliberately.)
